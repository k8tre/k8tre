# Keycloak post-install setup

When a fresh k8tre cluster comes up, the `keycloak-in-cluster` Argo CD
Application is Healthy and `https://keycloak.<domain>/` responds, but clicking
*Login with Keycloak SSO* on the portal lands on
`https://keycloak.<domain>/realms/k8tre-app/protocol/openid-connect/auth?…`
and Keycloak shows

> **k8tre-app — We are sorry…**
> Client not found.

This page documents why and what to run to fix it.

## What's missing

The realm `k8tre-app` is imported by
[`apps/keycloak/envs/dev/keycloak-realm-config.json`](../../apps/keycloak/envs/dev/keycloak-realm-config.json),
but the file contains **only realm-level configuration** (timeouts, client
scopes). No `clients`, no `users`, no `groups`, no realm roles. The OIDC
clients used by every k8tre app — `backend` (portal), `jupyterhub`, `gitea`,
`guacamole`, etc. — must be created separately.

The k8tre repo *does* contain manifests for these clients under
[`apps/keycloak/base/identity-management/keycloak.yaml`](../../apps/keycloak/base/identity-management/keycloak.yaml)
as `KeycloakClient` Custom Resources (group `identity.k8tre.io/v1alpha1`),
but:

- The CRDs are not yet shipped (see commit `ecb12b7 — Disable
  identity-management custom resources (CRDs not yet available)`), so the
  resources are commented out of `apps/keycloak/base/kustomization.yaml`.
- No controller in the cluster reconciles `KeycloakClient` resources into
  actual Keycloak clients via the Admin API, so even if the CRs were applied
  nothing would happen.

The repo ships [`ci/ci-setup-keycloak.py`](../../ci/ci-setup-keycloak.py)
which creates *only* the `jupyterhub` client (and a single demo user). The
other clients (`backend`, `gitea`, `guacamole`, `cr8tor`) have no automation
at all.

Net effect on a fresh install: the OIDC client secrets exist in the cluster
(they are auto-generated by `ci/create-ci-secrets.py` and synced via
ExternalSecret into each app namespace), but Keycloak itself has no client
registered with those credentials.

## Quick fix — create the clients via `kcadm.sh`

The Keycloak admin CLI is shipped inside the `keycloak-keycloakx-0` pod.
Run it through `kubectl exec`:

```sh
DOMAIN=<your-domain>          # e.g. 188.34.94.28.nip.io
ADMIN_USER=$(kubectl get secret -n keycloak keycloak-admin-credentials -o jsonpath='{.data.username}'      | base64 -d)
ADMIN_PWD=$( kubectl get secret -n keycloak keycloak-admin-credentials -o jsonpath='{.data.admin-password}' | base64 -d)

BACKEND_SECRET=$( kubectl get secret -n backend    backend-oidc-credentials    -o jsonpath='{.data.client-secret}' | base64 -d)
JHUB_SECRET=$(    kubectl get secret -n jupyterhub jupyterhub-oidc-credentials -o jsonpath='{.data.client-secret}' | base64 -d)

# Login (writes credentials to ~/.keycloak/kcadm.config inside the pod)
kubectl exec -n keycloak keycloak-keycloakx-0 -- \
  /opt/keycloak/bin/kcadm.sh config credentials \
    --server http://localhost:8080 --realm master \
    --user "$ADMIN_USER" --password "$ADMIN_PWD"

# Backend client — the OIDC client used by the portal app
kubectl exec -n keycloak keycloak-keycloakx-0 -i -- \
  /opt/keycloak/bin/kcadm.sh create clients -r k8tre-app -f - <<JSON
{
  "clientId": "backend",
  "name": "FastAPI Backend OIDC Client",
  "enabled": true,
  "protocol": "openid-connect",
  "publicClient": false,
  "standardFlowEnabled": true,
  "directAccessGrantsEnabled": true,
  "serviceAccountsEnabled": true,
  "clientAuthenticatorType": "client-secret",
  "secret": "$BACKEND_SECRET",
  "redirectUris": ["https://portal.$DOMAIN/auth/callback", "https://portal.$DOMAIN/*"],
  "webOrigins":   ["https://portal.$DOMAIN"],
  "attributes":   {"post.logout.redirect.uris": "+"}
}
JSON

# JupyterHub client
kubectl exec -n keycloak keycloak-keycloakx-0 -i -- \
  /opt/keycloak/bin/kcadm.sh create clients -r k8tre-app -f - <<JSON
{
  "clientId": "jupyterhub",
  "name": "JupyterHub OIDC Client",
  "enabled": true,
  "protocol": "openid-connect",
  "publicClient": false,
  "standardFlowEnabled": true,
  "clientAuthenticatorType": "client-secret",
  "secret": "$JHUB_SECRET",
  "redirectUris": ["https://jupyter.$DOMAIN/hub/oauth_callback"],
  "webOrigins":   ["https://jupyter.$DOMAIN"]
}
JSON
```

Repeat the same pattern for `gitea`, `guacamole`, `cr8tor` etc. when their
secrets land in the cluster — the `KeycloakClient` CRs under
`apps/keycloak/base/identity-management/keycloak.yaml` are the authoritative
reference for `redirectUris`, `webOrigins`, scopes and protocol mappers.

## Create a demo user

A working OIDC client is not enough — Keycloak rejects login without an
existing user. To unblock browser testing:

```sh
USER_ID=$(kubectl exec -n keycloak keycloak-keycloakx-0 -- \
  /opt/keycloak/bin/kcadm.sh create users -r k8tre-app \
    -s username=demo -s enabled=true \
    -s email=demo@example.com -s firstName=Demo -s lastName=User \
    -i 2>/dev/null)

kubectl exec -n keycloak keycloak-keycloakx-0 -- \
  /opt/keycloak/bin/kcadm.sh set-password -r k8tre-app \
    --userid "$USER_ID" --new-password demo
```

After this you can complete the SSO flow on the portal: click *Login with
Keycloak SSO* → Keycloak login form → `demo` / `demo` → redirect back to
`https://portal.<domain>/auth/callback?code=…`.

## Make the backend reach Keycloak from inside the cluster

After creating the clients and the user the SSO flow still does **not**
complete: the browser is redirected back to `/auth/callback?code=…` but the
portal home page keeps showing the *Login* button. Backend logs show

```
Auth callback error:
Error type: ConnectTimeout
File "/app/main.py", line 638, in auth_callback
    token = await oauth.keycloak.authorize_access_token(request)
File "...authlib/integrations/base_client/async_openid.py", line 62, in parse_id_token
    jwk_set = await self.fetch_jwk_set()
```

The cause is hard-coded in
[`ci/backend/main.py`](../../ci/backend/main.py) line 127:

```python
oauth.register(
    name="keycloak",
    ...
    access_token_url=f"{internal_base}/protocol/openid-connect/token",
    userinfo_url=f"{internal_base}/protocol/openid-connect/userinfo",
    jwks_uri=f"{external_base}/protocol/openid-connect/certs",   # ← external!
    issuer=external_base,
    ...
)
```

The token and userinfo endpoints can be redirected by setting
`KEYCLOAK_INTERNAL_URL`, but `jwks_uri` always resolves to
`https://keycloak.<public-host>/…`. From inside a pod the public hostname
resolves to the public IP, the kernel hairpins out the default route, hits
the cloud NAT, and the cloud refuses to loop traffic back to itself —
`ConnectTimeout` after ~10 s. Same for `issuer` validation.

### Fix: CoreDNS hosts override → VM host IP

Make every `*.<domain>` lookup from inside the cluster resolve to the VM's
primary private IP. Traffic then arrives at the host's `enp3s0:80/443`,
where the `socat` systemd units forward it to `cilium-envoy` on
`127.0.0.1:14722` — exactly the same path public Internet traffic takes.

The k3s default CoreDNS imports `*.server` files from a ConfigMap named
`coredns-custom` in `kube-system`:

```sh
VM_HOST_IP=10.30.1.61                # the VM's enp3s0 IP, not the public IP
DOMAIN=188.34.94.28.nip.io

kubectl apply -f - <<CM
apiVersion: v1
kind: ConfigMap
metadata:
  name: coredns-custom
  namespace: kube-system
data:
  nip-io.server: |
    $DOMAIN:53 {
      hosts {
        $VM_HOST_IP keycloak.$DOMAIN portal.$DOMAIN gitea.$DOMAIN \
                    jupyter.$DOMAIN guacamole.$DOMAIN cr8tor.$DOMAIN
        fallthrough
      }
    }
CM

kubectl rollout restart deployment coredns -n kube-system
```

(Use `*.server` and **not** `*.override`. `*.override` patches the existing
`.:53` block and produces `plugin/hosts: this plugin can only be used once
per Server Block` because k3s's stock Corefile already has a `hosts` plugin
for `/etc/coredns/NodeHosts`. `*.server` creates a separate server block
for the zone, which avoids the conflict.)

Verify from a workload pod:

```sh
kubectl exec -n backend deploy/backend -- python3 -c "
import urllib.request, ssl, socket
ctx = ssl.create_default_context(); ctx.check_hostname = False; ctx.verify_mode = ssl.CERT_NONE
print('dns:', socket.gethostbyname('keycloak.$DOMAIN'))
r = urllib.request.urlopen('https://keycloak.$DOMAIN/realms/k8tre-app/protocol/openid-connect/certs', context=ctx, timeout=8)
print('OK', r.status, len(r.read()))
"
# expect: dns: 10.30.1.61
#         OK 200 2933
```

### Approaches that don't work — and why

| Approach | What goes wrong |
|---|---|
| Set `keycloak_internal_url` to the cluster service URL (`http://keycloak-keycloakx-http.keycloak.svc:80`) | Fixes the token/userinfo endpoints but `jwks_uri` is built from `external_base` in the backend code, so JWKS fetch still hairpins. |
| Point DNS at the gateway service ClusterIP (`cilium-gateway-internal-gateway`) | Cilium represents that Service with the dummy endpoint `192.192.192.192` and the eBPF redirect to `cilium-envoy`'s loopback listener (`127.0.0.1:14722`) only works for traffic that originated on the host; from a pod the connection times out. |
| Point DNS at the keycloak service ClusterIP (`keycloak-keycloakx-http`) | Keycloak listens on **HTTP only** (`KC_HTTP_ENABLED=true`, no HTTPS keystore configured), so the implicit `:443` from the public URL returns *connection refused*. Adding a `443 → 8443` port alias on the Service doesn't help — `:8443` is a phantom port that nothing in the pod actually answers on. |
| iptables DNAT for traffic to the public IP from inside the pod | Cilium's `kubeProxyReplacement: true` runs eBPF programs at the TC ingress hook on `enp3s0` and on `cilium_host`, intercepting before iptables PREROUTING runs. The host-network workaround (the `socat` socket on `0.0.0.0:80/443`) is the only path that survives. |

### Backend ConfigMap caveat

A `backend-config` ConfigMap with the wrong URLs will be re-applied by
ArgoCD on its next reconcile if `selfHeal: true` is set on the
`backend-in-cluster` Application. The CoreDNS fix above is robust against
that — it lives in `kube-system` and isn't tracked by the backend
Application — so it is preferred over patching
`backend-config.keycloak_internal_url`.

## Verification

```sh
# 1) Keycloak HTTPS + realm reachable
curl -k -sI https://keycloak.<domain>/realms/k8tre-app/.well-known/openid-configuration | head -1
# expect: HTTP/1.1 200 OK

# 2) The auth URL the portal redirects to actually renders the login page
curl -k -s -o /tmp/auth.html \
  "https://keycloak.<domain>/realms/k8tre-app/protocol/openid-connect/auth?response_type=code&client_id=backend&redirect_uri=https%3A%2F%2Fportal.<domain>%2Fauth%2Fcallback&scope=openid+profile+email&state=test&nonce=test"
grep -o '<title>[^<]*</title>' /tmp/auth.html
# expect: <title>Sign in to k8tre-app</title>
```

If the auth URL responds with a Keycloak page titled "We are sorry... Client
not found.", the corresponding `client_id` is missing — re-run the `kcadm.sh
create clients` block above for that client.

```sh
# 3) Backend can fetch JWKS via the public hostname from inside the cluster
kubectl exec -n backend deploy/backend -- python3 -c "
import urllib.request, ssl
ctx = ssl.create_default_context(); ctx.check_hostname = False; ctx.verify_mode = ssl.CERT_NONE
r = urllib.request.urlopen('https://keycloak.<domain>/realms/k8tre-app/protocol/openid-connect/certs', context=ctx, timeout=8)
print(r.status, len(r.read()))
"
# expect: 200 2933
```

If this hangs, the CoreDNS hosts override is missing or pointing at the
wrong IP — re-do the *Fix: CoreDNS hosts override → VM host IP* step.

```sh
# 4) Complete browser flow
# Open https://portal.<domain>/ in a private window, click "Login with
# Keycloak SSO", enter demo/demo. The redirect should resolve in a few
# hundred milliseconds and land on /projects without the Login button.
```

## What's still missing for a complete setup

These are the moving parts that would make this section unnecessary; none of
them is currently in the repo:

1. **The remaining client registrations** — `gitea`, `guacamole`, `cr8tor`
   (the script in `ci/ci-setup-keycloak.py` only covers `jupyterhub`).
2. **Realm roles `k8tre-admins` / `k8tre-users`** plus the client-scope
   `k8tre-roles` that JupyterHub and the backend consume to authorize a
   user. `ci/ci-setup-keycloak.py` creates the scope but the realm roles
   are still TODO.
3. **The `groups` protocol mapper** on each client so the OIDC token
   includes the user's full group path — required by JupyterHub for the
   project-based namespacing.
4. **Group → project mappings**. The portal's `Project` CR
   (`research.k8tre.io/v1alpha1`, see
   `apps/keycloak/base/identity-management/projects.yaml`) currently has
   neither a CRD nor a reconciler. Until one ships, projects exist only in
   the static `projects-patch.yaml` file (and that file is what triggered
   [PR #6](https://github.com/eggai-tech/k8tre/pull/6), which had to disable
   the patch to make `kustomize build` succeed at all).

## Suggested upstream improvements

In rough order of impact:

- **Fix `ci/backend/main.py` line 127** to construct `jwks_uri` from
  `internal_base` (or take it as a separate `KEYCLOAK_JWKS_URL` env var).
  That removes the entire *Make the backend reach Keycloak from inside the
  cluster* section: with internal-only URLs there is no hairpin to work
  around, and the CoreDNS hosts override becomes unnecessary. The same
  applies to any other place where the backend hardcodes the external base
  for an in-pod call (e.g. `issuer` validation, end-session URL probing).
- **Make `agnostics/kare-dns` actually work.** The `dns-monitor` CronJob is
  meant to do exactly what the CoreDNS hosts override above does
  (rewriting `*.<domain>` to a cluster-internal endpoint), but its
  `monitor-dns.sh` shell script has an envsubst/sed escape bug
  (`s/\\${DOMAIN}/.../g` → `s/\188.34.94.28.nip.io/.../g` after the
  envsubst plugin runs → `sed: Invalid back reference`). Either fix the
  script or ship the CoreDNS hosts override as a static manifest in
  `bootstrap/`.
- **Extend `ci/ci-setup-keycloak.py`** to register every client the k8tre
  apps reference, reading the secret from the corresponding `*-oidc-credentials`
  Kubernetes Secret instead of taking it on the command line. Add a
  `kubectl apply -f` of this script as a one-shot `Job` so it runs as part
  of the GitOps reconciliation.
- **Document the post-install step** — the upstream
  [`docs/guides/installation.md`](../guides/installation.md) ends at "the
  apps are Healthy"; a short *Keycloak post-install* section pointing at
  this page would close the gap.
- **Ship a real `KeycloakClient` CRD + controller** so the manifests under
  `apps/keycloak/base/identity-management/` become declarative instead of
  documentation.
